Laboratorio #02 — NoSQL con Docker — I Cuatrimestre 2026

1. Introducción

El presente laboratorio tiene como objetivo principal explorar y comparar tres motores de bases de datos NoSQL ampliamente utilizados en la industria: Apache HBase, MongoDB y Redis. A través de la carga de un dataset real de comercio electrónico con más de 2,6 millones de registros, se evaluaron aspectos como el modelo de datos, la estrategia de carga, la ejecución de consultas nativas y el rendimiento medido en tiempo de respuesta.

Todo el entorno de ejecución fue montado utilizando Docker Compose, garantizando reproducibilidad y portabilidad entre sistemas operativos (Windows con WSL2 y Linux). Las consultas se implementaron tanto de forma nativa desde cada CLI como mediante scripts Python en cuadernos Jupyter, utilizando las bibliotecas happybase, pymongo y redis-py.

Objetivos

  • Desplegar HBase, MongoDB y Redis en contenedores Docker.
  • Cargar el dataset de historial de compras de electrónica en los tres motores.
  • Ejecutar consultas analíticas equivalentes en cada motor y medir tiempos de respuesta.
  • Comparar el modelo de datos, rendimiento y casos de uso de cada tecnología.

2. Motores de Base de Datos NoSQL

2.1 Apache HBase

HBase es una base de datos NoSQL de tipo columnar distribuida, inspirada en el modelo BigTable de Google. Está construida sobre el sistema de archivos distribuido HDFS (Hadoop Distributed File System) y forma parte del ecosistema Apache Hadoop. Su diseño está orientado a almacenar grandes volúmenes de datos semi-estructurados con baja latencia de lectura y escritura aleatoria.

Características principales

CaracterísticaDetalle
Modelo de datosTablas con filas identificadas por row key, organizadas en familias de columnas
EscalabilidadHorizontal, diseñada para clústeres de cientos de nodos
ConsistenciaFuerte (CP en el teorema CAP)
Lenguaje de consultaNo tiene SQL nativo; se accede vía API Java, Thrift o REST
Casos de uso típicosSeries de tiempo, logs, datos de sensores, historial de eventos
Imagen Dockerdajobe/hbase

En este laboratorio se utilizó el servidor Thrift (puerto 9090) para conectar Python mediante la biblioteca happybase. Los datos se organizaron en tres familias de columnas: event (timestamp), product (id, categoría, marca, precio) y user (id).

2.2 MongoDB

MongoDB es la base de datos NoSQL orientada a documentos más popular del mundo. Almacena los datos como documentos BSON (Binary JSON), lo que permite estructuras flexibles y anidadas sin esquema fijo (schema-less). Soporta un lenguaje de consulta rico (MQL) con operadores de agregación, índices secundarios, transacciones ACID multi-documento y replicación nativa.

Características principales

CaracterísticaDetalle
Modelo de datosDocumentos JSON/BSON anidados agrupados en colecciones
EscalabilidadHorizontal (sharding) y vertical
ConsistenciaConfigurable (CP por defecto, eventual en réplicas)
Lenguaje de consultaMQL + Aggregation Pipeline
Casos de uso típicosCatálogos, CMS, APIs REST, análisis en tiempo real
Imagen Dockermongo:7.0

Los datos se cargaron en la colección purchases de la base ecommerce. Se crearon índices en los campos category_code, brand y event_time para optimizar las consultas de agregación.

2.3 Redis

Redis (Remote Dictionary Server) es una base de datos NoSQL en memoria de tipo clave-valor, conocida por ser la más rápida del mercado para lecturas y escrituras. Aunque opera principalmente en RAM, ofrece persistencia opcional y soporta estructuras de datos avanzadas como listas, conjuntos, conjuntos ordenados (sorted sets), hashes y streams.

Características principales

CaracterísticaDetalle
Modelo de datosClave-valor con estructuras: string, hash, list, set, zset, stream
EscalabilidadCluster mode, replicación master-replica
ConsistenciaEventual (AP en el teorema CAP)
PersistenciaRDB (snapshots) y AOF (append-only file)
Casos de uso típicosCaché, sesiones, colas de mensajes, rankings en tiempo real
Imagen Dockerredis:7.2-alpine

La estrategia de carga utilizó Sorted Sets (ZSET) para almacenar rankings precalculados por categoría, marca e ingresos, permitiendo consultas de tipo "top-N" en tiempo sub-milisegundo. Los registros individuales se guardaron como HASH.

4. Configuración del Entorno (Docker)

4.1 Requisitos previos

  • Windows 10/11 con WSL2 habilitado + Docker Desktop, o Ubuntu 22.04 LTS con Docker Engine.
  • Python 3.10 o superior.
  • 8 GB de RAM mínimo recomendado (HBase consume ~1,5 GB).

4.2 Archivo docker-compose.yml

Se definieron tres servicios en una red interna labnet:

services:
  hbase:
    image: dajobe/hbase
    container_name: hbase
    ports: ["16010:16010", "9090:9090", "2181:2181"]
    networks: [labnet]

  mongodb:
    image: mongo:7.0
    container_name: mongodb
    ports: ["27017:27017"]
    environment:
      MONGO_INITDB_ROOT_USERNAME: admin
      MONGO_INITDB_ROOT_PASSWORD: admin123
    volumes: [mongo_data:/data/db]
    networks: [labnet]

  redis:
    image: redis:7.2-alpine
    container_name: redis
    ports: ["6379:6379"]
    command: redis-server --appendonly yes --maxmemory 2gb
    volumes: [redis_data:/data]
    networks: [labnet]

4.3 Comandos de gestión

  1. Levantar todos los servicios: docker compose up -d
  2. Verificar estado: docker compose ps
  3. Iniciar servidor Thrift de HBase (requerido para Python): docker exec -d hbase hbase thrift start
  4. Ver logs: docker compose logs <servicio>
  5. Detener sin borrar datos: docker compose stop
  6. Eliminar todo (reset): docker compose down -v
Importante: El servidor Thrift de HBase no inicia automáticamente con la imagen dajobe/hbase. Se debe ejecutar manualmente luego de levantar el contenedor y esperar ~15 segundos antes de conectar con Python.

5. Carga de Datos en los Tres Motores

Se implementaron scripts Python independientes para la carga en cada motor (load_hbase.py, load_mongodb.py, load_redis.py), además de un script unificado (load_data.py) que permite cargar los tres de forma simultánea. El CSV se procesó en chunks de 10.000 registros para controlar el uso de memoria.

5.1 Estrategia de carga en HBase

Se creó la tabla ecommerce con tres familias de columnas. Cada fila se identificó con la clave compuesta order_id_product_id. Se usó la API batch() de happybase para agrupar los puts en lotes de 1.000 operaciones. 

table_name = "ecommerce"
families   = {"event": {}, "product": {}, "user": {}}
conn.create_table(table_name, families)

# Por cada registro:
row_key = f"{row['order_id']}_{row['product_id']}".encode()
batch.put(row_key, {
    b"event:time"    : str(row["event_time"]).encode(),
    b"product:cat"   : str(row["category_code"]).encode(),
    b"product:brand" : str(row["brand"]).encode(),
    b"product:price" : str(row["price"]).encode(),
    b"user:id"       : str(row["user_id"]).encode(),
})

5.2 Estrategia de carga en MongoDB

Se utilizó insert_many(ordered=False) por chunks. Los timestamps se convirtieron a objetos datetime con zona UTC para permitir consultas de fecha con el Aggregation Pipeline. Los valores NaN de pandas se convirtieron a None (null en BSON). 

chunk["event_time"] = pd.to_datetime(chunk["event_time"], utc=True, errors="coerce")
docs = chunk.where(pd.notnull(chunk), None).to_dict("records")
collection.insert_many(docs, ordered=False)

5.3 Estrategia de carga en Redis

Se utilizó una arquitectura de multi-tipo: cada pedido se almacenó como un HASH (order:<id>), y se mantuvieron tres Sorted Sets precalculados que permiten responder las tres consultas analíticas en O(log N):

  • sales:by_category → score = cantidad de ventas acumulada por categoría
  • revenue:by_brand → score = ingresos totales acumulados por marca
  • sales:by_month → score = cantidad de ventas acumulada por mes (formato YYYY-MM)

Se usó un pipeline de Redis (pipeline(transaction=False)) para agrupar los comandos y reducir drásticamente el número de round-trips al servidor.

5.4 Tiempos de carga

MotorRegistros cargadosTiempo aprox.Velocidad aprox.
MongoDB2.633.521~74 segundos~35.500 reg/s
Redis2.633.521~120–180 segundos~15.000–22.000 reg/s
HBase2.633.521~25–40 minutos~1.100–1.700 reg/s
HBase es significativamente más lento en carga individual porque cada put implica una escritura en el WAL (Write-Ahead Log) y en MemStore. MongoDB es el más rápido gracias a su motor WiredTiger con compresión de escrituras en bloque. Redis tiene overhead por los múltiples comandos por registro (HSET + 3 ZINCRBY), pero el pipeline lo mitiga considerablemente.

  { $match: { brand: { $type: "string" } } },
  { $group: { _id: "$brand", revenue: { $sum: "$price" } } },
  { $sort: { revenue: -1 } },
  { $limit: 1 }
])
// → [ { _id: 'samsung', revenue: 90024615.05 } ]

db.purchases.aggregate([
  { $project: { month: { $dateToString:

  { $group: { _id: "$month", total: { $sum: 1 } } },
  { $sort: { total: -1 } },
  { $limit: 1 }

8.3 Análisis por motor

MongoDB

Demostró el mejor balance entre flexibilidad, velocidad de carga y rendimiento de consulta. El Aggregation Pipeline permite expresar consultas analíticas complejas de forma declarativa. Los índices secundarios redujeron significativamente el tiempo de las queries de agrupación. El principal desafío fue el manejo de valores NaN de pandas que se almacenan como double NaN en BSON en lugar de null, requiriendo filtrar con $type: "string".

Redis

Fue el motor más rápido en consulta con diferencia, respondiendo en microsegundos gracias a los Sorted Sets precalculados. Sin embargo, esto implica un diseño cuidadoso al momento de la carga: las estructuras de índice deben construirse durante la inserción. No es adecuado para consultas ad-hoc no anticipadas. Es ideal como capa de caché sobre datos ya agregados.

HBase

Fue el motor más lento tanto en carga como en consulta para este caso de uso. Al no tener un lenguaje de consulta nativo rico, las agregaciones requieren full table scans y procesamiento en el lado del cliente (Python), lo que lo hace inadecuado para análisis OLAP. Su fortaleza está en escrituras masivas a alta velocidad en entornos distribuidos de escala petabyte, como almacenamiento de logs o series temporales.

9. Conclusiones

La realización de este laboratorio permitió explorar en la práctica tres paradigmas distintos dentro del universo NoSQL, evidenciando que no existe un motor universalmente superior, sino que cada uno presenta ventajas y limitaciones según el tipo de carga de trabajo.

Sobre los modelos de datos: El modelo de documentos de MongoDB resultó el más natural para representar registros de transacciones, ya que cada documento puede contener todos los atributos de un pedido sin necesidad de definir un esquema previo. El modelo columnar de HBase, por su parte, es poderoso para datos de series temporales masivos, pero su API de bajo nivel requiere más código para tareas que en MongoDB son triviales. Redis obliga a pensar en las consultas que se necesitan antes de diseñar la estructura de datos, lo que implica un mayor esfuerzo de modelado pero produce el rendimiento más alto.

Sobre los tiempos de respuesta: La diferencia en consulta fue de varios órdenes de magnitud. Redis respondió en menos de 1 milisegundo, MongoDB en 1–4 segundos y HBase entre 25–35 segundos para el mismo tipo de pregunta. Esto corrobora que Redis es la elección ideal para rankings y dashboards en tiempo real, MongoDB para aplicaciones con consultas flexibles y HBase para ingestión masiva de datos en arquitecturas distribuidas.

Sobre el uso de Docker: Docker Compose simplificó enormemente el despliegue del entorno, permitiendo reproducir la misma configuración en diferentes sistemas operativos con un único comando. Se identificó un punto de atención importante: el servidor Thrift de HBase no inicia automáticamente, requiriendo un paso manual adicional que debe documentarse claramente.

Sobre la calidad de los datos: El dataset presentó un desafío real de ingeniería de datos: casi el 46% de los valores de category_code son nulos. El comportamiento diferente de pandas (float NaN), MongoDB (BSON double NaN) y las soluciones de filtrado requeridas ($type: "string") ilustra la importancia de entender cómo cada tecnología trata los valores faltantes antes de diseñar las consultas.

En conclusión, este laboratorio consolidó el conocimiento práctico sobre tres de los motores NoSQL más utilizados en la industria y demostró que la elección de la base de datos debe siempre estar guiada por los patrones de acceso, el volumen de datos, los requisitos de latencia y el modelo de consistencia necesario para cada caso de uso.
